# Task: Implement CACT (Content-Addressed Context Tree) in Floor

## Important: floor/ vs project_management/

`floor/` is the product — the template that ships to users. `project_management/` at the repo root is Floor using itself to manage its own development.

CACT is a feature being added to the Floor template. The new scripts, tree structure definition, and generated `cact_tree.json` all live inside `floor/` as part of the template. When a user copies Floor into their project and runs `cact_build.py`, it builds a CACT for *their* project.

The repo-root `project_management/` may eventually use CACT to manage Floor's own development, but that is NOT part of this task. This task builds the feature in `floor/`. Integration changes to repo-root files (like updating `floor/project_management/prompting.md` template) are about updating the template, not about changing how the Floor repo itself is managed.

Do not confuse these two directories. If you are about to modify a file at `project_management/scripts/floor.py` (repo root), stop and reconsider — you almost certainly mean `floor/project_management/scripts/floor.py` (the template).

## Overview

Replace Floor's manual cdoc authoring and flat routing table with an automated, tree-structured context system called CACT. The tree is built from source files, where leaf nodes are individual files and interior nodes hold LLM-generated summaries of their children. Content hashing at every level detects when summaries are stale and need regeneration.

Floor's enforcement mechanisms (CLAUDE.md rules, shutdown checks, plan-first gates, manifest auditing, architecture governance) stay intact. CACT replaces only the cdoc creation/maintenance workflow and the cdoc routing table.

## Key Concepts

### Node Types

Every node in the tree has a `node_type`. There are three:

1. **`source`** — A leaf node representing a source file (`.py`, `.md` prompt files, config files, data files). Contains no summary of its own; the file content *is* the content. Its `content_hash` is the SHA-256 of the file on disk.

2. **`directory`** — An interior node representing a group of children (usually corresponding to a filesystem directory, but grouping can be customized). Its `summary` is auto-generated by an LLM from its children's summaries/content. Its `content_hash` is the SHA-256 of its children's `content_hash` values concatenated in sorted key order. When `content_hash` changes (because a child changed), the summary is stale and must be regenerated.

3. **`explanation`** — A leaf node representing authored knowledge that is NOT auto-generated and must NOT be overwritten by regeneration. These are files like `architecture.md`, `style.md`, and `CLAUDE.md` — prescriptive documents that encode conventions, constraints, and design rationale. Their `content_hash` is the SHA-256 of the file, same as `source` nodes. The critical difference: when an `explanation` node's hash changes, it means a human or agent deliberately edited it. The system propagates the hash change upward (parent summaries may need regeneration to reflect the new conventions), but it never regenerates the explanation node itself.

Explanation nodes are the "why" layer. Source nodes are the "what" layer. Directory summaries bridge them — they describe what the children do, informed by any linked explanation nodes.

### Cross-References

A separate structure tracks lateral dependencies that don't fit the tree hierarchy. Each cross-reference is:
```json
{
  "source": "scripts/floor.py",
  "target": "standards/architecture.md",
  "relation": "enforced_by",
  "note": "Reads architecture.md for the arch gate check"
}
```
Cross-references are declared manually in the tree definition (not auto-detected in the MVP). When `cact_walk.py` produces context for a task, it appends cross-referenced nodes as "see also" items.

### Tree File: `cact_tree.json`

Generated file. Lives at `project_management/cact_tree.json` relative to the user's project root (i.e., wherever the user's project root is — NOT inside `../floor/` itself, since it's generated per-project). The `cact_build.py` script produces it.

Structure:

```json
{
  "version": 1,
  "root": "project",
  "nodes": {
    "project": {
      "node_type": "directory",
      "summary": "This project does X using...",
      "content_hash": "abc123...",
      "children": ["project/scripts", "project/standards"]
    },
    "project/scripts": {
      "node_type": "directory",
      "summary": "Python tooling for workflow orchestration...",
      "content_hash": "def456...",
      "children": ["project/scripts/floor.py", "project/scripts/shutdown.py"]
    },
    "project/scripts/floor.py": {
      "node_type": "source",
      "file_path": "project_management/scripts/floor.py",
      "content_hash": "ghi789..."
    },
    "project/standards/architecture.md": {
      "node_type": "explanation",
      "file_path": "project_management/standards/architecture.md",
      "content_hash": "jkl012..."
    }
  },
  "cross_references": [
    {
      "source": "project/scripts/floor.py",
      "target": "project/standards/architecture.md",
      "relation": "enforced_by",
      "note": "Checks arch gate via task counter threshold"
    }
  ]
}
```

Notes on the schema:
- `source` and `explanation` nodes have a `file_path` field (path relative to repo root on disk) and no `summary` or `children`.
- `directory` nodes have `summary` and `children` but no `file_path`.
- Node keys in the `nodes` dict are tree-path identifiers (e.g., `project/scripts/floor.py`), NOT filesystem paths. The `file_path` field on leaf nodes maps to the actual file on disk.
- The `root` field names the root node key.

## Tree Structure Definition

The tree structure is hardcoded in `cact_build.py` as a Python data structure. This is intentional — the grouping of files into a semantic tree is a deliberate design decision, not something to auto-detect from the filesystem. The hardcoded definition is the canonical mapping. Add a prominent comment in the code noting this is the place to update when the project structure changes.

The definition should be a dict or list that specifies, for each node: its key, its type, its children (for directories) or file_path (for leaves), and any cross-references. The exact data structure is up to you, but it must be easy for a human to read and edit.

For the MVP, the tree definition should describe a **generic Floor-managed project** — not specifically the Floor repo itself. The tree groups files the way a typical project using Floor would be organized. It should update setup.md to indicate that this template should be filled out with real project details on project start. Here is an example structure for a Floor project:

```
project (directory)
├── project/scripts (directory)
│   ├── project/scripts/floor.py (source) → project_management/scripts/floor.py
│   ├── project/scripts/shutdown.py (source) → project_management/scripts/shutdown.py
│   ├── project/scripts/check_manifest.py (source) → project_management/scripts/check_manifest.py
│   ├── project/scripts/check_cdocs.py (source) → project_management/scripts/check_cdocs.py
│   ├── project/scripts/check_cdoc_coverage.py (source) → project_management/scripts/check_cdoc_coverage.py
│   ├── project/scripts/task_counter.py (source) → project_management/scripts/task_counter.py
│   ├── project/scripts/hash_util.py (source) → project_management/scripts/hash_util.py
│   ├── project/scripts/cact_build.py (source) → project_management/scripts/cact_build.py
│   ├── project/scripts/cact_walk.py (source) → project_management/scripts/cact_walk.py
│   └── project/scripts/cact_update.py (source) → project_management/scripts/cact_update.py
├── project/standards (directory)
│   ├── project/standards/architecture.md (explanation) → project_management/standards/architecture.md
│   └── project/standards/style.md (explanation) → project_management/standards/style.md
├── project/governance (directory)
│   ├── project/governance/CLAUDE.md (explanation) → CLAUDE.md
│   ├── project/governance/manifest.md (explanation) → project_management/manifest.md
│   └── project/governance/prompting.md (explanation) → project_management/prompting.md
├── project/prompts (directory)
│   ├── project/prompts/architecture-check.md (source) → project_management/prompts/architecture-check.md
│   └── project/prompts/implement-this.md (source) → project_management/prompts/implement-this.md
└── project/status (directory)
    └── project/status/status.md (source) → project_management/status.md
```

Notes on this mapping:
- `CLAUDE.md`, `manifest.md`, and `prompting.md` are `explanation` — they contain prescriptive rules.
- `status.md` is `source` — it's data, not conventions.
- Prompt files (`architecture-check.md`, `implement-this.md`) are `source` — they're task artifacts.
- This tree does NOT artifacts, cdoc infrastructure, or generated data files (`cact_tree.json`, `cdoc_hashes.json`, `task_counter.txt`, `.floor_session.json`). Those are either generated or ephemeral.
- The user customizes this tree definition when they adopt Floor for their own project, adding their own source directories and files alongside the Floor scaffolding.

## Scripts to Implement

All new scripts go in `floor/project_management/scripts/`. Each script follows the existing Floor conventions: shebang + docstring, argparse, `--repo-root` flag, logic/IO/orchestration separation, exit codes 0/1/2. Python 3, stdlib only (no pip dependencies). LLM calls use subprocess to the `claude` CLI.

### 1. `cact_build.py`

**Purpose:** Bootstrap or fully rebuild `cact_tree.json` from scratch.

**Behavior:**
1. Read the tree structure definition (hardcoded in this script).
2. Walk all leaf nodes. For each, compute `content_hash` via `hash_util.file_sha256`.
3. For each directory node (bottom-up), compute `content_hash` as SHA-256 of sorted concatenation of children's `content_hash` values. Use the same aggregation pattern as `hash_util.compute_source_hash`.
4. For each directory node, generate a summary by calling the Claude CLI in non-interactive mode:
   ```
   claude -p "your prompt here" --model claude-sonnet-4-6
   ```
   Capture stdout as the summary text. The prompt should be:
   - For directory nodes whose children are all leaves: pass the actual file contents (or first ~200 lines if very large) and ask for a 3-5 sentence dense summary of what this group of files does, their dependencies, and constraints they enforce.
   - For directory nodes with directory children: pass the children's summaries and ask for a higher-level 2-4 sentence summary.
   - For directory nodes that have `explanation` children: follow the same process as leaves.
5. Write `project_management/cact_tree.json` (relative to repo root).
6. Print a report: number of nodes, number of summaries generated, total CLI calls.

Handle errors gracefully: if `claude` is not on PATH or the call fails, print an error and exit with code 2.

**Layer:** 2 — Check (same level as other diagnostic/build tools). Imports from `hash_util` (Layer 4) only.

### 2. `cact_walk.py`

**Purpose:** Given one or more node keys (or file paths), output the accumulated context from root to those targets.

**Behavior:**
1. Load `project_management/cact_tree.json`.
2. Accept arguments: one or more node keys or file paths. If file paths are given, resolve them to node keys by searching `file_path` fields. If `--all` is passed instead of specific keys, output the full summary tree (all directory node summaries, hierarchically formatted, plus a listing of all leaf nodes grouped by parent).
3. For each target leaf, collect all ancestor nodes from root to that leaf.
4. Merge ancestor paths (if two targets share a common ancestor, include the ancestor's summary only once).
5. Output the accumulated context to stdout as markdown:

```markdown
# Context Path

## project (root)
[root summary]

## project/scripts
[scripts summary]

### Target: project/scripts/floor.py
File: project_management/scripts/floor.py

---

## See Also
- project/standards/architecture.md (enforced_by): Checks arch gate via task counter threshold
  File: project_management/standards/architecture.md
```

6. The "See Also" section lists any cross-referenced nodes that are referenced by nodes in the context path but are not themselves in the path.
7. If `--full` flag is passed, also include the actual file contents of the target leaf nodes inline (read from disk). Without `--full`, list only file paths so the agent can read them itself.

**No CLI/API calls.** This script only reads `cact_tree.json` and optionally reads source files from disk. Pure I/O.

**Layer:** 2 — Check. Imports from `hash_util` only if needed.

### 3. `cact_update.py`

**Purpose:** Post-task script that detects changed files, regenerates stale summaries, and reports results.

**Behavior:**
1. Load `project_management/cact_tree.json`.
2. For every leaf node (`source` and `explanation`), recompute `content_hash` from the file on disk. Compare against stored hash.
3. If a leaf hash changed, mark it and all its ancestors as needing recomputation.
4. For stale `directory` nodes (process bottom-up), recompute `content_hash` from children. If changed, call `claude -p` to regenerate the summary (same prompting strategy as `cact_build.py`).
5. For `explanation` nodes whose hash changed: do NOT regenerate the explanation node itself. Just propagate the hash change upward so parent summaries get regenerated. Print a notice: `"explanation node updated: project/standards/architecture.md — parent summaries will be regenerated"`.
6. Check coverage: verify that all files in the repo (minus excludes) are represented as leaf nodes. Report any uncovered files.
7. Save updated `project_management/cact_tree.json`.
8. Print a structured report:
   - Which leaves changed (with file paths)
   - Which summaries were regenerated
   - Any uncovered files
   - Whether the root hash changed

**Modes:**
- Default: full update (recompute, regenerate, save).
- `--check-only`: recompute hashes and report staleness, but don't regenerate or save. Exit code 0 if fresh, 1 if anything is stale. (Used by `floor.py` pre-session.)
- `--no-api`: recompute hashes, update the tree file, but skip summary regeneration. Mark stale directory summaries with a `"stale": true` flag in the node dict. (For offline use or when `claude` CLI is unavailable.)

**Coverage excludes:** `.git`, `project_management/artifacts/`, `project_management/cact_tree.json`, `project_management/cdoc_hashes.json`, `project_management/task_counter.txt`, `.floor_session.json`, `.gitignore`, and everything in the .gitignore. Define as a constant list in the script + read .gitignore, similar pattern as `DEFAULT_EXCLUDES` in `check_cdoc_coverage.py`.

**Layer:** 2 — Check. Imports from `hash_util`. Makes CLI calls to `claude -p`.
